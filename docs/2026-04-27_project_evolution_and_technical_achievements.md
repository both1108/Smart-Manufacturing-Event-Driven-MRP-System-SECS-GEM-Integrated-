## 2. Technical Achievement Summary

| Technique / Pattern | Where It Is Used | Problem It Solves | Why It Matters in Real Manufacturing Systems |
|---|---|---|---|
| **Event-driven architecture** | `services/event_bus.py`, `services/domain_events.py`, every `services/subscribers/*.py` | Replaces polling + scattered side effects with a typed event stream that any number of subscribers can react to | A factory's reality is *events* (alarm fired, lot completed, recipe loaded), not snapshots. Polling produces stale dashboards and missed transitions; events produce correct ones. |
| **Event store (append-only)** | `services/event_store.py` (`event_store` table in `mysql/init.sql`); JSON encoding via `_encode` / `_decode`; rehydration via `latest_state_for(...)` | Provides a single durable, queryable, replayable record of *everything that ever happened* on every machine | This *is* the QA / recall record. When a customer reports a defective batch, the answer is `SELECT * FROM event_store WHERE correlation_id = ?` — equipment alarms, downtime, MRP decisions all in one timeline. |
| **Transactional outbox** | `services/event_store.py` (`append_many` writes `event_store + event_outbox` in one transaction); `services/outbox_relay.py` (single publisher) | Prevents the classic distributed-systems failure mode of "DB committed but message never published" (and vice versa) | A SECS S6F11 alarm must never be in the audit log without being delivered to MRP — and must never be delivered to MRP without being in the audit log. The transactional outbox makes that an invariant, not a hope. |
| **`FOR UPDATE SKIP LOCKED`** | `EventStore.fetch_undispatched(...)` in `services/event_store.py` | Lets multiple relay processes drain the same outbox in parallel without each grabbing the same row | The line never stops. You should be able to deploy a new app version with rolling restart while the relay keeps draining. SKIP LOCKED is what makes that safe. |
| **In-process pub/sub** | `services/event_bus.py` (`EventBus.publish`, MRO-aware dispatch including base-class subscribers as a "firehose") | Decouples publishers from subscribers without standing up Kafka or Redis on day one | At demo / single-line scale this is the right tool — µs latency, deterministic ordering, trivial test setup. The interface (`bus.publish` / `bus.subscribe`) is identical to what you'd expose over Redis Streams later, so migration is a single-file change. |
| **Per-machine actor model** | `services/machine_actor.py` (`MachineActor` with one mailbox per tool), `services/machine_actor_registry.py`, `services/ingest.py` (routes signals to actors) | Eliminates cross-machine races — every signal *and* every operator command for one tool is serialized through one queue, one consumer | Without this, a STOP click and a stale RUN telemetry sample can race the FSM into "M-01 transitioned RUN→IDLE→RUN→ALARM" in the wrong order, corrupting the audit trail. The actor model is what guarantees per-machine causality. |
| **Finite State Machine (per-tool)** | `services/state_machine.py` — `{IDLE, RUN, ALARM, UNKNOWN}` with `advance(...)` returning a `TransitionResult` of `[StateChanged, AlarmTriggered, AlarmReset]` | One place — and only one place — decides what state transitions are valid and what events they emit | Equipment vendors model GEM states (E30) explicitly. Mirroring that in code stops "alarm without state change" and "state change without event" bugs that haunt ad-hoc string comparisons. |
| **Transitional tailer + adapter swap** | `services/machine_data_tailer.py` (Phase 6), `services/secs/host_adapter.py` (Phase 7), `SIGNAL_SOURCE` flag in `bootstrap.py` | Migrates the system from a database-tailer transport to real SECS/GEM HSMS without rewriting downstream code | Real factory rollouts are *gradual* — you can't replace the data path of a running line in one go. The seam at `EquipmentIngest.offer(...)` lets the transport be swapped behind a flag, run in parallel, then the legacy side removed. |
| **Debounced MRP recomputation** | `services/subscribers/mrp_recompute_scheduler.py` (`threading.Timer` per part, in-memory `_pending` dict, 5 s debounce window) | A burst of alarms on machines that all produce PART-A becomes one MRP recompute, not N races | MRP is the most expensive subscriber in the pipeline. Without debouncing, every alarm storm becomes a thundering-herd of recomputes against the same `mrp_plan_view` row. With it, MRP work is bounded by a window, not by alarm rate. |
| **Two-phase MRP (projected vs reconciled)** | `MRPRecomputeScheduler._on_alarm_triggered` (reason=`projected_loss`, MTTR-based) and `_on_downtime_closed` (reason=`reconciled_loss`, actual lost qty) | Decouples *prediction at alarm time* from *correction at recovery time* | Operations needs an early signal ("we may need to reorder PART-A by Wednesday") *and* an authoritative one once the machine recovers. The two recomputes share a `correlation_id`, so the timeline is auditable end-to-end. |
| **CQRS read models** | `services/subscribers/read_model_projector.py`, `alarm_projector.py`, `telemetry_projector.py`; tables `machine_status_view`, `mrp_plan_view`, `mrp_plan_history` in `mysql/init.sql`; `services/query/*` reads them | Separates the slow, append-only write path (events) from the fast, denormalized read path (views) | The dashboard query "show me current state for all 50 tools" must be O(machines), not O(events_ever). Projectors maintain that view incrementally; the dashboard never joins the audit log live. |
| **Correlation ID tracing** | `DomainEvent.correlation_id` (default `uuid4`); `MRPRunner._on_recompute` propagates `ev.correlation_id` to `MRPPlanUpdated`; `MRPRecomputeScheduler` chains it from the originating alarm; `correlation_id` columns on `equipment_events`, `machine_downtime_log`, `mrp_plan_history` | Lets a single ID link every event in a causal chain across equipment, production, and business layers | This is the audit answer to "why did we order 800 units of PART-A on Tuesday?" → JOIN on `correlation_id` → equipment alarm at 10:03 on M-02 → downtime closed 47 minutes later → projected loss 320 units → reconciled loss 410 → MRP recompute → recommended PO. One JOIN, full causal trace. |
| **DLQ + retry semantics** | `services/event_store.py` (`mark_failed`, `move_to_dlq`, `event_dlq` table), `services/outbox_relay.py` (`MAX_ATTEMPTS=5`, `_maybe_dlq`) | Stops a poison event from looping forever; preserves the row for human inspection | A malformed payload from a vendor SECS handler can't take down the line. The relay isolates it, the rest of the pipeline keeps moving, ops can replay later. |
| **SECS/GEM integration** | `services/secs/host_adapter.py` (one `EquipmentSession` per machine, parallel `enable()` / `disable()` lifecycle), `services/secs/session.py`, `services/secs/decoders.py`, `services/secs/config.py`, `simulators/secs_equipment/` | Speaks the actual fab-floor protocol (HSMS over TCP, S1/S5/S6 messages) instead of inventing a custom one | Every real semiconductor / display / electronics line speaks SECS/GEM. Without this layer the project is "interesting Python," not "deployable in a fab." With it, swapping the simulator container for a real tool is a config change. |
| **Capacity-adjusted MRP** | `services/subscribers/capacity_tracker.py` computes `lost_qty = duration_hours × nominal_rate × efficiency`; `services/subscribers/mrp_impact_handler.py` writes `capacity_loss_daily`; `services/mrp_service.simulate_inventory_and_mrp(...)` consumes it | The MRP simulation isn't run against forecast alone — it's run against forecast minus *actual capacity loss caused by this morning's downtime* | This is the bridge between equipment reality and procurement decisions. A static MRP that ignores tool downtime is a planning fiction. The capacity-adjusted version is what ops can actually trust. |
| **Idempotent ingestion** | `services/ingest.py` (`RawEquipmentSignal.edge_seq`, dedup window in `EquipmentIngest._seen`) | Restarting the tailer or retransmitting a SECS message doesn't double-process the underlying signal | Both the database tailer and the SECS handler can replay rows on restart. Idempotency at the ingestion edge means downstream subscribers (FSM, MRP) never see duplicates. |

## 3. Current System Capability (end-to-end)

What you can actually demo today by running `docker compose up` and clicking through the React dashboard:

**Machine signals enter the system** through one of two interchangeable transports — the legacy IoT path (`iot_simulator.py` writes to `machine_data`, `services/machine_data_tailer.py` polls by AUTO_INCREMENT id and offers `RawEquipmentSignal` into `EquipmentIngest`) or the SECS/GEM path (`simulators/secs_equipment/` speaks HSMS on ports 5001/5002/5003, `services/secs/host_adapter.py` decodes S6F11 messages and offers the same `RawEquipmentSignal` into the same ingest queue). Both paths converge at `EquipmentIngest.offer(...)` — the seam that makes equipment transport pluggable.

**Equipment state changes** are inferred per-machine. `EquipmentIngest` routes each signal to the actor for that `machine_id` (`services/machine_actor_registry.py`), the actor reads the metrics, the FSM in `services/state_machine.py` advances `from_state → to_state` and emits `StateChanged` events. Every transition is durable in `event_store` *before* the in-memory state advances.

**Alarms are triggered and resolved** through the same FSM. `AlarmTriggered` events carry `alid`, `alarm_text`, and `metrics`; `AlarmReset` events carry `previous_state` and `resolved_to`. The `AlarmProjector` maintains an alarm read model; `services/query/alarms_query.py` exposes it via `routes/alarms_routes.py`. Operator-initiated overrides (Phase 11) flow through the same actor mailbox as `ControlAction`, producing `HostCommandRequested → HostCommandDispatched | HostCommandRejected` events.

**Capacity degradation** is computed by `services/subscribers/capacity_tracker.py`. Leaving RUN for ALARM/IDLE opens a row in `machine_downtime_log`; returning to RUN closes it and computes `lost_qty = duration_hours × nominal_rate × efficiency` (rate/efficiency from `machine_capacity`). A `DowntimeClosed` event is emitted with the loss, the affected `produces_part`, and the original `correlation_id`.

**MRP is recomputed** through a debounced two-phase pipeline. `MRPRecomputeScheduler` listens to `AlarmTriggered` (emits `projected_loss` after a 5 s window using MTTR + nominal rate) and to `DowntimeClosed` (emits `reconciled_loss` with actual lost qty). Each recompute writes an `MRPRecomputeRequested` row to `event_store` (so it goes through the same outbox/DLQ machinery as equipment events). `MRPRunner` subscribes to that, calls the pure `simulate_inventory_and_mrp(...)` function with capacity loss subtracted from `incoming_qty`, persists per-day breakdown to `mrp_plan_history`, and emits `MRPPlanUpdated` carrying the summary.

**Material shortages are detected** inside `simulate_inventory_and_mrp(...)` (`services/mrp_service.py`). `MRPPlanUpdated.has_shortage`, `total_shortage_qty`, and `earliest_shortage_date` are first-class fields on the event.

**Purchase recommendations** are computed in the same simulation: `suggested_po_qty` and `suggested_order_date` (lead-time-aware) are populated on `MRPPlanUpdated` and projected into `mrp_plan_view` by `ReadModelProjector`. *(Caveat: there is currently no Postgres-side projector that turns this into a business-system row — see §4.)*

**The dashboard / API surface** reads exclusively from projected views. `services/query/dashboard_query.py`, `events_query.py`, `machines_query.py`, `alarms_query.py`, and `command_service.py` expose JSON over `/api/*` routes; the React app under `Frontend/` consumes them. CORS is scoped to `/api/*` only, so `/healthz` / `/readyz` operator probes stay same-origin.

**Traceability is one SQL JOIN long.** Because every event in the chain (alarm → downtime → recompute → plan) shares a `correlation_id`, the chain `AlarmTriggered → DowntimeClosed → MRPRecomputeRequested → MRPPlanUpdated` is reconstructable from `event_store` alone.

## 4. Known Weaknesses / Next Upgrade Targets

Identified in `SECSGEM/architecture_review_2026-04-27.md`. Listed in
order of operational severity.

**Slow subscribers run on the relay thread.** `OutboxRelay._drain_once` (`services/outbox_relay.py`) calls `self._bus.publish(event)` synchronously inside `asyncio.to_thread`. `EventBus.publish` (`services/event_bus.py`) walks the MRO and invokes every handler inline. So `MRPRunner._on_recompute` — which does pandas DataFrame work plus multiple MySQL round-trips inside the simulation — runs *on the relay's drain thread*, blocking dispatch of every other event for every other machine until it finishes. A 400 ms recompute on PART-A delays an `AlarmTriggered` for M-07 by 400 ms. The fix is per-subscriber decoupling (see §5) — fast subscribers like `ReadModelProjector` should never sit behind slow ones.

**`EventBus.publish` swallows subscriber exceptions; the outbox marks the event dispatched anyway.** `event_bus.py` catches and logs every handler exception. The relay then sees a clean return from `bus.publish` and calls `mark_dispatched`. Failure mode: `CapacityTracker.on_state_changed` raises `pymysql.err.OperationalError` → log line → outbox row marked dispatched → `DowntimeClosed` is never written → MRP never sees the loss → purchase suggestion is wrong, silently. The audit trail in `event_store` looks perfect. Fix: either re-raise after logging (lets the existing relay retry/DLQ cover handler failures) or add per-subscriber acknowledgment via a `event_subscriber_offsets` table.

**`MachineHeartbeat` is stored in `event_store`.** `services/machine_actor.py` emits a `MachineHeartbeat` every 2 s per machine even when the FSM hasn't transitioned — one INSERT into `event_store` plus one into `event_outbox`, just to give the UI a fresh metric sample. At 3 demo machines, ~1.5 rows/s. At a 50-tool line: ~2.16 M rows/day, ~780 M rows/year, mostly identical metric snapshots. The state-change audit is drowned in heartbeats and the outbox's `FOR UPDATE SKIP LOCKED` scan walks a much longer undispatched window. Heartbeats are a *telemetry stream*, not a *domain event* — they should land in a separate `machine_telemetry` table (or a time-series store on the Postgres side via `TelemetryProjector`), not in the audit log.

**Debounce state is in-memory.** `services/subscribers/mrp_recompute_scheduler.py` keeps `self._pending: Dict[str, _Pending]` and `self._timers: Dict[str, threading.Timer]` per process. Two app instances both subscribe to `AlarmTriggered`, both run independent debouncers, both emit `MRPRecomputeRequested` after 5 s — with **different** `correlation_id`s. The outbox's row-level locking dedupes publication, not creation. Fix: persist the debounce window in MySQL (`pending_recompute(part_no PK, expires_at, last_correlation_id)`) and let any instance claim the recompute via `UPDATE ... WHERE expires_at < NOW()`.

**Dual-database / procurement projector is missing.** `requirements.txt` declares `psycopg2-binary` and `docker-compose.yml` boots a Postgres instance for the business side. `grep -rn "psycopg2\|postgres" services/` returns nothing — every subscriber writes to MySQL only. `MRPPlanUpdated` carries `suggested_po_qty` and `suggested_order_date` but no subscriber translates that into a row in any business-side table. The bounded-context split is asserted in comments and `requirements.txt`; the wire isn't built. The fix is a `services/subscribers/procurement_projector.py` listening on `MRPPlanUpdated` and writing to a Postgres `purchase_recommendations(part_no, suggested_qty, suggested_date, correlation_id, source_alarm_correlation)` table. Until that exists, "equipment → business traceability" is a half-claim.

**Other latent issues** documented in the review: no event schema versioning (`_decode` does `cls(**raw)` blind, so adding a field to `AlarmTriggered` will break replay of old events); no DLQ admin endpoint or alerting; `flask` + `asyncio` running in one process is fragile under `gunicorn --workers N`; no Prometheus metrics on `events_published_total`, `outbox_lag_seconds`, `dlq_depth`, `mrp_recompute_duration_seconds`.

## 5. Recommended Next Upgrade

> **Move `MRPRunner` off the relay thread by introducing a persisted `mrp_command_queue` table and a single-consumer worker that drains it with `FOR UPDATE SKIP LOCKED`.**

This is the single highest-leverage change available this week.

**Why a persisted `mrp_command_queue`.** Today `MRPRecomputeRequested` reaches `MRPRunner` via `bus.publish` on the relay thread. Adding an in-memory queue would fix the threading problem but lose durability — a crash between "scheduler decided to recompute" and "runner finished" would silently drop the recompute. Persisting the command in MySQL makes the decision durable the moment the scheduler's subscriber returns. The schema mirrors the outbox: `(id, part_no, reason, triggered_by, correlation_id, source_event_seq, enqueued_at, picked_at, picked_by, finished_at, attempts, last_error)`. Same retry semantics, same DLQ logic, same ops mental model.

**Why move `MRPRunner` into an independent worker.** The relay thread's job is *fanout* — it must stay fast so dispatch latency for *every* machine stays low. A pandas-driven simulation has no business sharing that thread. An independent `MRPWorker` (`asyncio.create_task` running `claim_batch → simulate → mark_done`) lets the slow path scale on its own clock. It also restores the "single publisher = relay" invariant: the worker writes `MRPPlanUpdated` *to* `event_store` (using `EventStore.append_many`), the relay picks it up and publishes it, and downstream subscribers (`ReadModelProjector`, the future `ProcurementProjector`) react exactly as they do today. Nothing downstream of the relay changes.

**Why `FOR UPDATE SKIP LOCKED` on the queue.** It is the same primitive that makes `event_outbox` multi-instance-safe; reusing it on the new queue means a second app replica ships safely. Without it, two workers would race on the same `mrp_command_queue` row, both run the simulation, both emit `MRPPlanUpdated` with the same correlation, and the dashboard would briefly show two plans for one part. With `SKIP LOCKED`, each row is claimed by exactly one worker, and a worker crash leaves the row reclaimable after a recovery sweep on `picked_at IS NOT NULL AND finished_at IS NULL AND picked_at < NOW() - INTERVAL X MINUTE`.

**Side-effects this also fixes.** Persisting the command upstream of the simulator partially fixes the in-memory debounce concern in §4: even if the scheduler in two replicas both emit `MRPRecomputeRequested`, the queue can collapse adjacent `(part_no, reason)` rows by inserting with `ON DUPLICATE KEY UPDATE` on a `(part_no, reason, NOT finished)` partial unique constraint — converting "two recomputes" into "one recompute, idempotent re-enqueue." It also moves the project decisively toward the multi-instance deployment model needed for any real factory rollout.

**What it does *not* attempt to fix this week.** No Prometheus metrics yet (do them on top of *both* outbox and the new queue at once — same shape). No Postgres `procurement_projector` yet (build the bridge after the queue lands; the queue is upstream prerequisite for safe MRP runs). No bus replacement (Redis Streams / Kafka) — see §6 of `architecture_review_2026-04-27.md`; in-process is correct at this scale, and the queue is the right intermediate.

**Concrete deliverables for the upgrade.** Schema for `mrp_command_queue` in `mysql/init.sql`. New `repositories/mrp_command_repository.py` with `enqueue`, `claim_batch`, `mark_done`, `mark_failed`. New subscriber `services/subscribers/mrp_command_enqueuer.py` taking the `MRPRecomputeRequested` slot that `MRPRunner` currently holds. New worker `services/mrp_worker.py` with `start()` / `stop()` lifecycle matching `OutboxRelay`. `services/mrp_runner.py` refactored to expose `run_for(part_no, reason, triggered_by, correlation_id)` instead of `_on_recompute(ev)`. `bootstrap.py` swap: replace `mrp_runner.register(bus)` with `mrp_enqueuer.register(bus)` + `mrp_worker.start()`. The full code sketch lives in §9 of the 2026-04-27 architecture review.

The acceptance test for the upgrade is single-line: trigger an alarm storm on three machines that all produce PART-A; assert that **dispatch latency for `AlarmTriggered` events on unrelated machines does not regress**, while MRP recomputes still complete and `MRPPlanUpdated` still carries the original alarm's `correlation_id`. That test is the difference between "event-driven" and "event-driven *and operationally correct*."
